## Context

目前系統已實現基本的購物與結帳流程，訂單資料儲存在 PostgreSQL 中。現有的訂單相關 API 主要面向普通用戶，用於查看其個人訂單歷史。本設計旨在擴展系統，為管理員提供全局訂單管理能力，包括跨用戶查詢、狀態調整及詳細資訊檢視。

## Goals / Non-Goals

**Goals:**
- **全局查詢能力**：管理員可查看、搜尋與篩選所有用戶的訂單。
- **狀態受控轉移**：管理員可手動更新訂單狀態（如：待出貨 -> 已出貨）。
- **權限隔離**：確保只有具備 `admin` 權限的人員可訪問相關 API 與前端頁面。
- **資料完整性**：記錄狀態變更的時間與操作者（若適用）。

**Non-Goals:**
- **自動化金流對接**：本階段不涉及與第三方支付平台的自動退款 API 對接，僅提供狀態標記。
- **自動化庫存補回**：取消訂單後的庫存釋放暫由現有機制處理，不在本設計範圍內進行大幅重構。

## Decisions

- **API 路徑設計**：
  - 決定採用 `/api/v1/admin/orders` 作為根路徑，以符合現有的 API 版本化與權限隔離規範。
  - **理由**：遵循專案已有的 `/api/v1` prefix 慣例，簡化權限中間件配置，並明確 API 用途。
  
- **權限控制 (Security)**：
  - 必須通過 `current_active_admin` 相依注入檢查，驗證用戶的 `is_admin` 標籤。
  - **理由**：利用現有的 Auth 模組，確保安全性。

- **資料模型變更**：
  - 在 `OrderModel` 新增 `admin_note` (Text) 欄位，用於管理員內部分記。
  - 新增 `status_updated_at` (DateTime) 記錄最後一次狀態變更時間。

- **前端技術選型**：
  - 使用 `shadcn/ui` (Radix Vue) 的 `DataTable` 組件來實現列表，支援分頁與客戶端/伺服器端過濾。
  - **理由**：與專案現有的後台介面風格一致，開發效率高。

## Risks / Trade-offs

- **[Risk] 狀態誤操作** → [Mitigation] 在更新狀態前提供二次確認彈窗，並限制狀態跳轉的合法性（例如：已取消訂單不可改為已完成）。
- **[Trade-off] 大數據量查詢性能** → [Decision] 強制分頁，並針對常用搜尋欄位（如訂單編號、用戶 ID）建立資料庫索引。
