# 功能規格書：商品管理模組 (Product Management Module)

**功能分支**: `010-product-module`
**建立日期**: 2026-02-12
**狀態**: 草稿
**輸入**: 使用者描述: "建立商品模組。"

## 使用者情境與測試 *(必填)*

### User Story 1 - 管理商品 (優先級: P1)

作為一名商店管理員，我希望能在系統中新增、更新和刪除商品，以便維護最新的商品目錄供顧客瀏覽。

**優先級理由**: 這是商店營運的基礎；沒有商品就沒有商店。這使得「商品目錄」功能得以動態運作。

**獨立測試**: 驗證透過 API/管理介面建立的商品是否出現在商品列表中，並可透過 ID 檢索。

**驗收情境**:

1. **Given** 我是一名已驗證的管理員, **When** 我提交有效的商品詳細資訊（名稱、價格、描述、庫存）, **Then** 系統建立一個新商品並回傳其唯一 ID。
2. **Given** 一個現有的商品, **When** 我更新其價格或庫存數量, **Then** 變更被持久化並反映在隨後的檢索中。
3. **Given** 一個現有的商品, **When** 我請求刪除它, **Then** 該商品從活躍目錄中移除（或軟刪除），且不再能透過標準公開查詢檢索到。

---

### User Story 2 - 管理分類 (優先級: P2)

作為一名商店管理員，我希望能將商品分類，以便顧客能輕鬆找到他們正在尋找的商品。

**優先級理由**: 隨著商品數量增加，這對於組織目錄是必要的。

**獨立測試**: 驗證分類可以被建立並指派給商品。

**驗收情境**:

1. **Given** 我是一名已驗證的管理員, **When** 我建立一個新分類（例如「電子產品」）, **Then** 它被儲存並可用於指派。
2. **Given** 一個商品和一個分類, **When** 我將該商品指派給該分類, **Then** 當依該分類篩選時，該商品會出現。

---

### User Story 3 - 商品圖片管理 (優先級: P3)

作為一名商店管理員，我希望能將圖片與商品關聯，以便顧客能看到他們正在購買什麼。

**優先級理由**: 視覺效果對電子商務轉換率至關重要，儘管系統最初可以在技術上使用佔位符運作。

**獨立測試**: 驗證圖片 URL 或檔案參照可以隨商品儲存。

**驗收情境**:

1. **Given** 一個商品, **When** 我提供一個有效的圖片 URL 或上傳一張圖片, **Then** 商品記錄被更新以包含該圖片參照。

---

### 邊界案例 (Edge Cases)

- **無效資料**: 嘗試建立價格為負數或名稱為空的商品應回傳驗證錯誤 (400 Bad Request)。
- **重複分類**: 嘗試建立名稱重複的分類應被阻止或優雅地處理。
- **並發更新**: 兩位管理員同時更新同一商品應被處理（例如：最後寫入者勝或版本檢查）。

## 需求 *(必填)*

### 功能需求 (Functional Requirements)

- **FR-001**: 系統必須提供用於建立、讀取、更新和刪除 (CRUD) 商品的 API 端點。
- **FR-002**: 系統必須提供用於建立、讀取、更新和刪除 (CRUD) 商品分類的 API 端點。
- **FR-003**: 系統必須驗證商品輸入資料：
    - 名稱: 必填，非空。
    - 價格: 必填，非負數。
    - 庫存: 整數，非負數。
- **FR-004**: 系統必須允許將一個商品指派給一個或多個分類。
- **FR-005**: 系統必須支援在管理介面（API 層級）中分頁列出商品。
- **FR-006**: 系統必須支援商品的軟刪除，以保留歷史訂單資料（如適用）。
- **FR-007**: 系統必須保護這些管理端點，僅允許經授權的管理員存取。

### 關鍵實體 (Key Entities) *(若功能涉及資料則包含)*

- **Product (商品)**:
    - `id`: 唯一識別碼。
    - `name`: 字串，必填。
    - `description`: 文字，選填。
    - `price`: 十進位數，必填。
    - `stock_quantity`: 整數。
    - `is_active`: 布林值（用於隱藏商品而不刪除）。
    - `category_id`: 關聯至 Category。

- **Category (分類)**:
    - `id`: 唯一識別碼。
    - `name`: 字串，必填，唯一。
    - `slug`: 字串，URL 友善的唯一識別碼。

### 假設與依賴 (Assumptions & Dependencies)

- **假設**:
    - 已存在供管理員使用的認證系統（會員系統）。
    - 基本的圖片託管策略已決定（例如：本地存儲或雲端 bucket）。
- **依賴**:
    - 依賴 User/Auth 模組進行管理員驗證。

## 成功標準 (Success Criteria) *(必填)*

### 可測量結果 (Measurable Outcomes)

- **SC-001**: 建立商品的 API 回應時間低於 500ms。
- **SC-002**: 列出商品（分頁，20 筆項目）的 API 回應時間低於 200ms。
- **SC-003**: 100% 的無效商品建立請求（例如：負價格）皆被拒絕並回傳適當的錯誤代碼。
- **SC-004**: 資料一致性：建立的商品可立即透過 ID 檢索。
