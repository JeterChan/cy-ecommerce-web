# 功能規格書：訂單結帳與管理系統

**功能分支**: `020-order-checkout-system`  
**建立日期**: 2026-03-15  
**狀態**: 草稿  
**輸入**: 使用者描述：「優化 Order Module, 開發以下兩件事情： 1. 結帳與建立訂單：讀取使用者的 Redis 購物車，核對最新價格與庫存，扣除實體庫存，並產生一筆新訂單。開發著重在“資料庫交易”，確保扣除商品庫存、建立訂單紀錄、建立訂單明細都被包裝在同一個資料庫交易中。這三個動作必須“全部成功”或“全部失敗”。 2. 查看個人訂單，讓使用者結帳完畢後，可以查看訂單狀態，需要包含所有訂單的資訊，包含訂單明細、收件人資訊、付款方式、寄送地址等。」

## 使用者情境與測試 *(強制性)*

### 使用者故事 1 - 事務性結帳與訂單建立 (優先級: P1)

作為一名購物車內已有商品的顧客，我希望能夠安全地完成購買，以便我能收到商品，同時確保庫存和訂單記錄的準確性。

**為什麼優先級為此**: 這是核心的營收流程。確保資料完整性（原子性交易）對於防止庫存差異和損壞的訂單至關重要。

**獨立測試**: 可以透過啟動包含多個項目的結帳流程來進行測試。當庫存被正確扣除、訂單記錄存在且訂單明細與購物車匹配，且所有操作同時生效時，即驗證成功。若失敗（例如模擬流程中途崩潰），則應導致庫存無變動且無訂單記錄。

**驗收場景**:

1. **假定** 我的 Redis 購物車中有 2 件商品，**當** 我點擊「下單」時，**那麼** 系統檢查最新價格/庫存，從每件商品的實體庫存中扣除 1，建立一筆「訂單」記錄，並在單個交易中建立 2 筆「訂單明細」記錄。
2. **假定** 購物車中有一件商品庫存不足，**當** 我嘗試結帳時，**那麼** 交易必須失敗，不扣除任何庫存，不建立任何訂單，且我收到「庫存不足」的錯誤。
3. **假定** 系統在扣除庫存後但在建立訂單記錄前崩潰，**當** 資料庫交易處理該失敗時，**那麼** 庫存扣除必須自動回滾。

---

### 使用者故事 2 - 查看個人訂單歷史與詳情 (優先級: P2)

作為一名已完成購買的顧客，我希望能夠查看我的訂單狀態和詳情，以便追蹤我的遞送進度並保留交易記錄。

**為什麼優先級為此**: 對於客戶服務和透明度至關重要。它允許使用者驗證其購買並查看其商品的所在地。

**獨立測試**: 可以透過以現有訂單使用者的身份登入並導航至「我的訂單」頁面來進行測試。當所有訂單資訊（項目、收件人、付款、地址）準確顯示時，即驗證成功。

**驗收場景**:

1. **假定** 我過去曾下過 3 筆訂單，**當** 我查看訂單歷史時，**那麼** 我會看到這 3 筆訂單的列表及其摘要資訊（訂單 ID、日期、總計、狀態）。
2. **假定** 我點擊特定的訂單 ID，**當** 詳情頁面載入時，**那麼** 我會看到完整的收件人姓名、送貨地址、所選付款方式，以及所有購買項目的清單（名稱、價格、數量、小計）。

---

### 邊界情況 (Edge Cases)

- **價格變動**: 如果在加入購物車和點擊「下單」之間產品價格發生變動會發生什麼？系統必須阻擋結帳並通知使用者價格已變動，要求他們在繼續之前確認新價格。
- **並行結帳**: 多個使用者同時結帳最後一件庫存商品。系統必須確保庫存不會低於零。
- **購物車過期**: 使用者嘗試結帳已在 Redis 中過期的購物車。
- **付款失敗**: 系統處理訂單已建立但外部付款流程失敗的情況（例如：狀態保持為「待付款」）。

## 需求 *(強制性)*

### 功能需求

- **FR-001**: 系統必須在結帳初始化期間從 Redis 快取中檢索使用者目前的購物車項目。
- **FR-002**: 系統必須在進行交易前，針對持久化產品資料庫進行產品可用性和單價的即時驗證。
- **FR-003**: 系統必須在單個資料庫交易中執行以下三項操作：
    - 扣除每個購買產品的實體庫存數量。
    - 建立新的訂單記錄（表頭）。
    - 建立對應的訂單項目記錄（明細）。
- **FR-004**: 系統必須保證「全有或全無」的原子性；如果 FR-003 中的任何操作失敗，整個交易必須回滾。
- **FR-005**: 系統必須在初始版本中支援貨到付款 (COD)。
- **FR-006**: 系統必須允許經過身份驗證的使用者查詢其自身的訂單歷史並查看任何特定訂單的完整詳情。
- **FR-007**: 系統必須追蹤並顯示訂單狀態。訂單生命週期必須遵循：處理中 (Processing) -> 已出貨 (Shipped) -> 已完成 (Completed) / 已取消 (Cancelled)。
- **FR-008**: 系統必須為每筆訂單持久化收件人資訊（姓名、電話）、送貨地址和付款方式詳情。

### 關鍵實體 *(若功能涉及數據則包含)*

- **訂單 (Order)**: 代表最高層級的購買記錄。包含使用者 ID、狀態、總金額、收件人資訊、送貨地址、付款方式和時間戳記。
- **訂單項目 (OrderItem)**: 代表訂單中的特定產品行。包含訂單 ID、產品 ID、購買時的價格和數量。
- **產品 (Product)**: 庫存水準和目前定價的單一事實來源。
- **購物車 (Cart)**: 基於 Redis 的臨時儲存，存放使用者打算購買的項目。

## 成功標準 *(強制性)*

### 可衡量結果

- **SC-001**: 100% 已完成的結帳導致零庫存差異（實體庫存 == 初始庫存 - 已售數量）。
- **SC-002**: 在發生系統或資料庫故障時，保持資料完整性，且訂單或訂單項目出現「孤兒」記錄的機率為 0%。
- **SC-003**: 對於擁有最多 50 筆訂單的帳戶，使用者能在 1 秒內檢索其訂單歷史列表。
- **SC-004**: 訂單詳情頁面在 1.5 秒內載入，並提供所有 5 個強制性數據點（項目、收件人、付款、地址、狀態）。

## 假設

- **身份驗證**: 使用者必須登入才能結帳和查看訂單。
- **庫存**: 庫存在產品層級（或 SKU 層級）進行管理。
- **購物車**: Redis 購物車包含 `product_id` 和 `quantity`。
- **並行處理**: 將使用資料庫行級鎖定或樂觀鎖定來處理並行庫存扣除。
