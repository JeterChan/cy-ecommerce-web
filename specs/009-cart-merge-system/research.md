# 研究報告: 後端購物車合併系統

**功能**: 購物車合併系統
**狀態**: 完成

## 技術決策

### 1. 資料儲存策略

**決策**:
- **訪客購物車**: 僅儲存於 **Redis**，設定存活時間 (TTL，例如 7 天)。
- **會員購物車**: 儲存於 **PostgreSQL** (用於持久化) 並快取於 **Redis** (用於效能)。
- **Redis 結構**: 使用 Redis **Hash** 資料型態。
  - Key: `cart:{user_id}` (會員) 或 `cart:{guest_token}` (訪客)。
  - Field: `product_id`。
  - Value: `quantity` (整數)。

**理由**:
- **Redis**: 提供低延遲存取 (<200ms 目標)，這對電商購物車互動至關重要。
- **Hash Type**: 允許高效修改個別商品 (O(1))，無需取出/序列化整個購物車。
- **PostgreSQL**: 確保會員購物車資料不會因 Redis 清空或崩潰而遺失，滿足「永久保存」需求。
- **訪客 TTL**: 自動清理廢棄的購物車。

**考慮過的替代方案**:
- *Redis JSON*: 適合結構化資料，但對於僅需 ID-數量 對應的情境，Hash 更簡單。
- *僅使用 Redis (AOF/RDB)*: 資料遺失風險高於 SQL；對購物車資料進行查詢/分析較困難。
- *僅使用 SQL*: 對於高頻率更新 (例如快速點擊 "+" 按鈕) 效能較差。

### 2. 訪客識別

**決策**: 產生一個 UUID v4 作為 `guest_token`，儲存於通用的 HTTP-only Cookie 中。

**理由**:
- 與登入 Session 機制解耦。
- UUID v4 具備統計上的唯一性。
- Cookie 儲存允許跨瀏覽器重啟後保留 (直到過期)。

### 3. 合併邏輯 (訪客轉會員)

**決策**: 「合併並累加 (Merge and Sum)」策略。
- 當使用者登入時:
  1. 從 Redis 讀取訪客購物車。
  2. 從 DB (或 Redis 快取) 讀取會員購物車。
  3. 對於訪客購物車中的每項商品:
     - 若會員購物車已存在: `會員數量 += 訪客數量`。
     - 若不存在: 新增至會員購物車。
  4. 將更新後的會員購物車儲存至 DB 並更新 Redis 快取。
  5. 從 Redis 刪除訪客購物車。
  6. 清除 `guest_token` cookie (可選，或直接忽略)。

**理由**:
- 對使用者最直觀；不會遺失商品。
- 保留了「過去會員工作階段」與「當前訪客工作階段」的意圖。

### 4. 實作技術棧

**決策**:
- **語言**: Python 3.12 (使用者要求)。
- **Web 框架**: FastAPI。
- **快取**: Redis。
- **ORM**: SQLAlchemy (Async)。

**理由**: 符合現有專案技術棧與使用者限制。

## 未知與釐清事項

### 已解決

- **Redis 角色**: 「快取儲存」暗示存在持久化儲存。我們將使用現有的 PostgreSQL DB 進行會員持久化。
- **商品資料**: 購物車僅參照 `product_id`。詳細商品資訊 (名稱、價格) 將在執行時從商品服務/資料表取得 (Join 或分開讀取)，以確保價格即時性。

### 假設

- `Product` 資料表存在或可被 Mock。
- `User` 資料表存在。
- 庫存檢查發生在結帳時，而非加入購物車時 (簡化購物車邏輯)。
