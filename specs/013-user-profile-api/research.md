## 1. 電子郵件驗證策略

### 背景
更新電子郵件需要驗證新舊電子郵件地址，以確保安全性和有效性。

### 決策
- **機制**: 使用 Redis 儲存臨時驗證權杖 (Tokens)。
- **流程**:
    1. 使用者請求變更電子郵件 -> 產生 2 個權杖（一個用於舊電子郵件，一個用於新電子郵件）。
    2. 發送包含這些權杖連結的電子郵件。
    3. 使用者點擊連結 -> API 驗證權杖。
    4. 當兩個權杖都經過驗證（在時間限制內），電子郵件將在資料庫中更新。
- **過期時間**: 權杖在 24 小時後過期。

### 理由
- **安全性**: 防止攻擊者在獲得帳戶存取權但未獲得電子郵件存取權的情況下劫持帳戶。
- **效能**: Redis 非常適合用於短期、高頻率存取的資料，如驗證權杖。
- **簡單性**: 避免使用臨時權杖使主資料庫變得混亂。

### 考慮過的替代方案
- **資料庫資料表**: 持久但較慢；需要清理工作。
- **無狀態 JWT**: 可行，但如果需要使其失效則較困難。

## 2. 帳戶刪除策略（延遲硬刪除）

### 背景
帳戶應立即被「軟刪除」（停用），然後在 30 天後被「硬刪除」（資料移除）。

### 決策
- **機制**: 「軟刪除」標記 + 每日 Cron Job（透過 Celery Beat）。
- **流程**:
    1. 使用者請求刪除 -> 設定 `is_active=False`，`deleted_at=Now`。
    2. 每日 Celery 任務執行：`DELETE FROM users WHERE deleted_at < (Now - 30 days)`。
- **權杖失效**: 立即將目前的更新/存取權杖列入黑名單（使用 Redis 黑名單或權杖版本控制）。

### 理由
- **穩健性**: 每日執行工作比安排 30 天後的單一任務更可靠（因為如果佇列重置，該任務可能會遺失）。
- **簡單性**: 易於實作與監控。

### 考慮過的替代方案
- **Celery 延遲任務**: `delete_account.apply_async(args=[user_id], countdown=30*24*60*60)`。如果 broker/result backend 被清除，有遺失任務的風險。
- **資料庫觸發器 (Database Trigger)**: 邏輯隱藏在資料庫中，難以維護和版本控制。

## 3. 技術堆疊與架構

### 背景
專案使用 Python、FastAPI 和 Clean Architecture。

### 決策
- **語言**: Python 3.12 (與 `011-order-module` 一致)。
- **框架**: FastAPI。
- **架構**: Clean Architecture (分層架構)。
    - **展示層 (Presentation)**: API Routers, DTOs (Pydantic)。
    - **應用層 (Application)**: Use Cases (業務邏輯)。
    - **領域層 (Domain)**: 實體 (Entities), 儲存庫介面 (Repository Interfaces)。
    - **基礎設施層 (Infrastructure)**: 儲存庫實作 (SQLAlchemy), 電子郵件服務, Redis 客戶端。
- **資料庫**: PostgreSQL (透過 SQLAlchemy)。
- **任務佇列**: Celery (用於電子郵件發送和刪除任務)。

### 理由
- 與現有的專案標準和限制保持一致。
- 確保可維護性和可測試性。
