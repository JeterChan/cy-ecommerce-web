# 功能規格書：購物車 Redis 優化與庫存驗證

**功能分支**: `018-cart-redis-optimization`
**建立日期**: 2026-03-13
**狀態**: 草案
**輸入**: 使用者描述: "根據現有的專案，將哪些商品的資訊儲存在 Redis? 由於商品資訊是經常需要存取的，因此儲存在 Redis 可以增加讀取效率，但是並不是將所有的商品資訊都儲存在 Redis, 只需要儲存 product_id, quantity。每次使用者查看購物車時，後端必須拿著 product_id 去 PostgreSQL 撈取最新價格來動態計算，如此才能防止價格不同步的漏洞。 此外，使用者將商品加入購物車，不能超過庫存，加入前必須去 PostgreSQL 檢查該商品的 stock 是否充足。"

## 使用者場景與測試 *(必填)*

### 使用者故事 1 - 包含庫存檢查的加入購物車 (優先級: P1)

作為一名顧客，我想將商品加入我的購物車，我希望系統能確保我加入的數量不會超過目前的庫存量。

**為什麼這是高優先級**: 這是防止超賣並確保購物車與庫存之間數據完整性的核心功能。

**獨立測試**: 可以透過嘗試加入特定數量的商品，並驗證系統是否根據資料庫中的庫存正確接受或拒絕請求來進行測試。

**驗收場景**:

1. **假設** 某商品有 10 個庫存，**當** 使用者加入 5 個到購物車時，**那麼** 購物車應在 Redis 中更新，包含 `product_id` 且 `quantity: 5`。
2. **假設** 某商品有 2 個庫存，**當** 使用者嘗試加入 5 個到購物車時，**那麼** 系統應拒絕請求並顯示「庫存不足」訊息。

---

### 使用者故事 2 - 包含動態定價的查看購物車 (優先級: P1)

作為一名顧客，我想在我的購物車中看到最新的價格，這樣如果在我加入商品後價格發生變化，我也不會被錯誤計費。

**為什麼這是高優先級**: 這對於防止價格相關的漏洞至關重要，並確保顧客看到的始終是當前的市場價格。

**獨立測試**: 可以透過將商品加入購物車，在資料庫中更改其價格，然後重新整理購物車頁面以查看更新後的總計來進行測試。

**驗收場景**:

1. **假設** 我的 Redis 購物車中已有一個商品，**當** 我查看購物車時，**那麼** 系統應從資料庫中獲取該 `product_id` 的最新價格並動態計算總計。

---

### 邊緣案例

- **併發庫存變動**: 如果在庫存檢查和最後更新購物車之間庫存減少了怎麼辦？（假設：檢查發生在更新前夕，以最小化此時間窗）。
- **結帳期間的價格變動**: 如果使用者在購物車頁面時價格發生變化，系統如何處理？（要求：重新整理或進入結帳流程時必須重新驗證價格）。

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系統在 Redis 購物車儲存中，核心業務數據僅存儲 `product_id` 和 `quantity`（可包含如時間戳記等必要的元數據，但不包含價格或商品名稱等應由資料庫提供的資訊）。
- **FR-002**: 系統在允許將商品加入或在 Redis 購物車更新數量之前，必須在 PostgreSQL 資料庫中執行實時庫存檢查。
- **FR-003**: 系統不得允許任何商品的購物車數量超過資料庫中的可用 `stock_quantity` 值。
- **FR-004**: 每當檢索購物車或計算總計時，系統必須從 PostgreSQL 資料庫獲取購物車中每項商品的當前 `price`。
- **FR-005**: 如果由於庫存不足而無法加入商品，系統必須向使用者提供明確的反饋（如 HTTP 400 錯誤與具體訊息）。

### 關鍵實體 *(如果功能涉及數據則包含)*

- **購物車項目 (Redis)**: 包含 `product_id`（識別碼）和 `quantity`（整數）以及時間戳記等基本狀態的輕量級記錄。
- **商品 (PostgreSQL)**: `price`（十進制）和 `stock`（整數）的真理來源。

## 成功標準 *(必填)*

### 可衡量的成果

- **SC-001**: 從 Redis 檢索 ID 和數量的延遲應低於 50ms（針對典型的購物車大小）。
- **SC-002**: 100% 的購物車總計計算在請求時使用當前的資料庫價格。
- **SC-003**: API 不允許出現任何「超賣」情況（購物車數量 > 資料庫庫存）。
- **SC-004**: 在「加入購物車」操作期間，使用者應在 200ms 內收到庫存驗證反饋。
