# 功能規格書：會員購物車非同步同步系統 (Redis + PostgreSQL)

**功能分支**: `019-member-cart-async-sync`  
**建立日期**: 2026-03-13  
**狀態**: 草案  
**輸入**: 使用者描述: "我希望會員採用 Redis+PostgreSQL 的方式儲存購物車，並且會員將商品加入購物車時，先更新 Redis, 爭取更快得讀寫，讓使用者可以馬上得知是否有無添加商品至購物車。再透過 Redis Stream 發送通知，背景程式讀取到 Redis Stream 的通知後，再將購物車的內容更新到 postgreSQL。"

## 使用者場景與測試 *(必填)*

### 使用者故事 1 - 快速加入購物車 (優先級: P1)

作為一名登入會員，我想快速地將商品加入購物車，並立即在 UI 上看到更新，而不需要等待資料庫事務完成。

**為什麼這是高優先級**: 提供即時的反饋感是提升使用者體驗的關鍵，尤其是在高併發情況下。

**獨立測試**: 可以透過觀察「加入購物車」API 的響應時間，以及檢查 Redis 緩存是否立即更新來測試。

**驗收場景**:

1. **假設** 會員已登入，**當** 將商品加入購物車時，**那麼** 系統應立即更新 Redis 緩存並回傳成功響應。
2. **假設** 會員在短時間內連續點擊加入不同商品，**當** 查看購物車時，**那麼** 系統應從 Redis 快速讀取並顯示所有已加入的商品。

---

### 使用者故事 2 - 數據最終一致性同步 (優先級: P1)

作為系統管理員，我希望會員的購物車數據最終能安全地儲存到 PostgreSQL 中，以確保長期持久性。

**為什麼這是高優先級**: 確保數據不會因為 Redis 揮發性或意外重啟而丟失。

**獨立測試**: 可以在 API 回傳成功後，稍等數秒檢查 PostgreSQL 資料庫中的 `cart_items` 表是否已自動更新。

**驗收場景**:

1. **假設** 使用者成功將商品加入 Redis，**當** 背景背景同步程式運行時，**那麼** 系統應透過 Redis Stream 獲取變更通知並正確寫入 PostgreSQL。

---

### 邊緣案例

- **Redis Stream 訊息遺失**: 如果 Redis 在訊息發送後、消費前崩潰怎麼辦？ (假設：使用 Redis 持久化 AOF/RDB 並結合 Stream 的消費確認機制)。
- **同步失敗處理**: 如果 PostgreSQL 寫入失敗（如唯一鍵衝突或斷線），背景程式將執行自動重試（3-5 次），若持續失敗則將訊息移入死信隊列 (DLQ) 以供人工干預。
- **併發衝突**: 為了確保跨裝置操作的順序正確，系統將透過 Redis 分散式鎖確保**單一會員**的變更訊息按產生的先後順序序列化寫入資料庫，防止不同步任務之間的競態條件。

## 需求 *(必填)*

### 功能性需求

- **FR-001**: 系統必須將會員購物車同時儲存於 Redis (快取層) 與 PostgreSQL (持久層)。
- **FR-002**: 會員「寫入」操作（新增、更新、刪除）必須優先更新 Redis 並立即回傳響應。
- **FR-003**: 每次 Redis 更新後，系統必須向指定的 Redis Stream 發送一則變更訊息。
- **FR-004**: 背景消費程式必須訂閱 Redis Stream，並將變更內容非同步寫入 PostgreSQL。
- **FR-005**: 購物車「讀取」操作應優先從 Redis 讀取，若 Redis 缺失則回退到 PostgreSQL 並回填 Redis。
- **FR-006**: 同步過程必須保證等冪性（Idempotency），透過訊息重試機制與資料庫鎖定防止重複消費導致數據錯誤。

### 關鍵實體 *(如果功能涉及數據則包含)*

- **購物車快取 (Redis Hash)**: 儲存 `product_id`, `quantity` 及最後更新時間戳。
- **同步訊息 (Redis Stream)**: 包含 `user_id`, `action_type`, `payload` 的事件流。
- **購物車實體 (PostgreSQL)**: 持久化儲存的購物車項目。

## 成功標準 *(必填)*

### 可衡量的成果

- **SC-001**: 會員加入購物車 API 的 P95 響應時間應低於 30ms。
- **SC-002**: 99.9% 的購物車變更應在 2 秒內同步至 PostgreSQL。
- **SC-003**: 系統應能處理 Redis 暫時失效的情況，並在恢復後自動續傳未完成的同步訊息。
- **SC-004**: 使用者在不同分頁刷新購物車時，看到的內容應具備高度的一致性（延遲感極低）。
